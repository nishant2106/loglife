# Log Life Product Scope

## Product Summary

`Log Life` is a personal life-logging application for people who want to record their day, memories, and events in a simple and searchable way.

The core idea is:

- a user creates a log by typing or speaking
- the system converts that input into structured text
- the system extracts useful entities from the log
- those entities are stored separately but remain linked back to the original log
- the user can later search, browse, and revisit their life through dates, people, events, and other connected information

For now, this is a personal product for a single user mindset, but it should be built in a way that can later support multiple users signing up and keeping their own private data.

## Product Goals

- make life logging easy and low friction
- support both text-first and voice-first input
- help users remember people, events, moods, and incidents across time
- make past information retrievable even when the exact date is forgotten
- create a calm, simple, mindfulness-oriented product experience
- keep the system ready for LLM-based enrichment, extraction, and summarization workflows

## V1 Product Direction

Version 1 should focus on the personal logging experience, not on social or collaboration features.

The first version should include:

- lightweight authentication
- text and voice-based log creation
- date-based and time-based logging
- structured extraction of key entities from logs
- search across logs and linked entities
- basic analytics around user activity
- support for attachments such as images and files

The product should also be designed with future LLM usage in mind:

- log understanding will rely heavily on LLM-based processing
- summaries, extraction, and structuring may use model APIs
- users may later bring their own API key
- the platform may later provide a default API key and charge based on token usage

Billing and API key management are not part of the first build, but the architecture should leave room for:

- per-user LLM configuration
- tracking model usage per operation
- usage accounting for future billing
- switching between platform-managed and user-managed model credentials

The first version should not include:

- notifications
- calendar integrations
- advanced sharing or collaboration
- complex wellness recommendations

## Primary User

The primary user is an individual who wants to maintain a private personal record of their life without needing to memorize everything.

This user wants to:

- log what happened during a day
- record memories or events that may span a date range
- revisit people and incidents mentioned in past logs
- search for something even if they do not remember the exact date

## Core Logging Model

The `log` is the primary record in the system.

A log can represent:

- a full day summary
- a specific event
- a memory
- an activity within a day
- a date range memory or multi-day event

Each log should support:

- a single date
- a start date and end date for date-range logs
- optional start time and end time for time-specific logs
- raw user input
- processed text generated by the system
- structured extracted entities

## Input Modes

Users should be able to create a log in two ways:

- by typing in chat-like or form-like input
- by speaking and letting the app transcribe the audio

Voice logging flow for V1:

- user records audio
- system stores the audio file
- system transcribes speech into text
- system uses the transcript as the source for a structured log
- system extracts entities and links them to the log

Text logging flow for V1:

- user types a log
- system stores the raw input
- system creates a cleaned or structured version if needed
- system extracts entities and links them to the log

## Entity Extraction Model

After a log is created, information identified inside the log should be stored in its own entity tables and linked back to the source log.

Initial entity groups:

- `people`
- `events`
- `goals`
- `moods`
- `attachments`

Expected behavior:

- a user can open a log and see the linked people, events, goals, and mood references
- a user can open a person and see all related logs and incidents
- a user can open an event and see all linked logs and related context

This means the system should treat the log as the source document, while entities become structured reference objects derived from that source.

## Search Expectations

Search is a core product feature.

Users should be able to:

- search by words or phrases from past logs
- find the day or date range when something happened
- search for a person and see their related logs
- search for an event and see the logs connected to it

V1 search priority:

- full-text search over logs
- search over extracted people and event names
- returning linked logs and dates as part of the result

Advanced semantic or RAG-style search is not required in the first build.

## Analytics Expectations

V1 analytics should stay lightweight and practical.

Initial analytics may include:

- number of logs created
- active days logged
- streaks or consistency of logging
- number of people or events identified
- simple activity trends over time

## Privacy and Access

For V1, all user data should be private by default.

Authentication should be simple initially:

- user name and email for identity
- later support email verification, OTP, or magic link style authentication

Each user should only be able to access their own logs, entities, and files.

## UI Direction

The product should feel:

- simple
- mindful
- calm
- easy to begin using

The home screen should be minimal and centered around the primary action:

- start a new log
- choose text or voice
- quickly attach a date, date range, or time range

The rest of the product should support browsing and revisiting:

- logs
- people
- events
- analytics

## Proposed Core Data Model

Initial core tables or models:

- `users`
- `logs`
- `log_inputs`
- `people`
- `events`
- `goals`
- `moods`
- `attachments`
- `log_people`
- `log_events`
- `log_goals`
- `log_moods`

Important notes:

- `logs` should support date and date-range storage
- `logs` should support optional time-range storage
- extracted entities should remain linkable back to the source log
- all user-generated records should support soft delete

## First Build Priorities

The first few features should likely be:

- create a log with text input
- create a log with voice input and transcription
- list and search logs
- extract and link people from logs
- open a person detail page and see linked logs

This gives the product a usable end-to-end loop:

1. capture life
2. structure it
3. retrieve it later

## Open Decisions

These still need to be finalized before implementation starts:

- exact authentication flow for V1
- whether entity extraction is automatic only or user-confirmed
- whether events are always extracted or can also be manually created
- exact attachment limits and supported file types
- exact home screen layout and visual design language
- exact analytics widgets for V1
